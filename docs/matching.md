# Matching engine — algorithm, taxonomy, and limitations

This documents the deterministic keyword-coverage matcher (contract v2). The
normative contract is [ADR 0001](adr/0001-privacy-first-demo-contract.md); the
canonical implementation is `backend/app/matching.py`, mirrored byte-for-byte
by the TypeScript port and gated by the shared fixtures in
`backend/tests/fixtures/contract_cases.json`.

**What this is:** an honest, transparent keyword-coverage check — "which of the
job description's recognised skill keywords appear, non-negated, in the
resume?" **What it is not:** a measure of candidate quality, seniority, or
real-world fit. It cannot read depth, recency, or context.

## Algorithm

1. **Skill extraction.** Both texts are scanned against a compact dictionary
   (`backend/app/skills.json`) of canonical skills and their aliases. Matching
   is case-insensitive with token-aware boundaries: `js` matches as a word but
   not inside `next.js`; a trailing sentence period does not break a match.
   A skill counts at most once no matter how often it is mentioned (keyword
   stuffing does not help).
2. **Tier classification (JD side, first).** Each JD skill is classified
   `required` or `nice_to_have` from section headers ("Nice to have:",
   "Requirements:") and inline cues ("is a plus", "preferred", …), scoped per
   sentence/line. The default is the safe `required`; downgrading needs every
   mention to sit in a nice-to-have context.
3. **Negation filter (resume side).** The resume is split into segments (line
   breaks + sentence terminators). A segment matching any negation /
   non-experience pattern — "No X experience", "never used", "not familiar",
   "lacks experience", "eager/plan/want to learn", "currently learning", … (the
   exact regex list lives in the ADR and in `NON_EVIDENCE_PATTERNS`) —
   contributes **neither skill presence nor evidence**. A skill still counts if
   a different, non-negated segment mentions it.
4. **Scoring.** `fit_score = round(100 * |required ∩ resume| / |required|)`
   using Python's default banker's rounding (12.5 → 12, 37.5 → 38 — identical
   in the TS port). Nice-to-have coverage is reported separately and never
   enters the denominator.
5. **Status.** If the recognised required list is empty (no JD skills at all,
   or all recognised skills are nice-to-have) the result is
   `insufficient_signal`: `fit_score` 0 (not meaningful, must not be rendered
   as a score), no suggestions, fixed explanatory summary. Nice-to-have
   coverage is still reported when present.
6. **Evidence.** Each matched skill maps to the first non-negated resume
   segment that mentions it (collapsed, capped at 200 chars).
7. **Suggestions.** Deterministic templates composed in the core (≤5, exact
   order and wording in the ADR). They deliberately never encourage adding a
   keyword the candidate cannot support ("If you *genuinely* have X
   experience…", "Only add one if you can support it truthfully").

Summary bands: `Strong coverage` (fit ≥ 80), `Partial coverage` (≥ 50),
`Low coverage` (else).

## Taxonomy decisions (skills.json)

`backend/app/skills.json` is the single source of truth;
`frontend/lib/skills.json` is generated by `scripts/sync-skills.mjs` (CI fails
on drift). Guiding rules: an alias must denote the *same* skill, not a broader
concept; no alias may map two unrelated tools to one canonical skill; and no
single text mention may count twice through parent/alias relationships.

Changes made for v2, with rationale:

| Change | Rationale |
| --- | --- |
| Split `html/css` → `html` (`html`, `html5`) and `css` (`css`, `css3`, `scss`, `sass`) | HTML and CSS are separate skills; a JD asking for both should be able to miss one. |
| Split `pandas` → `pandas` and `numpy` | Distinct libraries; NumPy on a resume must not satisfy a pandas requirement (and vice versa). |
| `terraform`: removed `iac`, `infrastructure as code` | Generic concept, not the tool — Pulumi/CloudFormation are also IaC. "Terraform for IaC" still matches via the tool name. |
| `kafka`: removed `pub/sub`, `event-driven` | Generic architecture concepts; "Pub/Sub" is also a distinct GCP product. |
| `docker`: removed `containerization`, `containers` | Conflated podman/containerd/LXC (and generic phrasing) with Docker specifically. |
| `observability`: removed bare `logging`, `tracing`; added `distributed tracing` | Every developer "logs"; "tracing" alone collides with e.g. ray tracing. `monitoring` kept: in resume/JD context it is a reliable observability signal. |
| `git`: removed `github`, `gitlab` | Hosting platforms are not the version-control skill; also fixed a double count where one "GitHub Actions" mention yielded both `ci/cd` and `git`. `version control` kept as a near-universal synonym. |
| `node.js`: removed `express` | English-word false positives ("express ideas") and a distinct framework besides. |
| `serverless`: `lambda` → `aws lambda` | Bare "lambda" matches Python lambda expressions. |
| `microservices`: removed `distributed systems` | Strictly broader concept (a monolithic distributed database is not microservices). |

Kept deliberately (documented judgment calls):

- `llm`: vendor/product names `openai`, `anthropic`, plus `rag`, `gpt`. In a
  resume context these are used almost exclusively to describe hands-on LLM
  work; treating them as LLM-skill signals is the pragmatic reading. They all
  map to the single canonical `llm`, so no double counting.
- `kubernetes`: `k8s`, `eks`, `gke` — managed-Kubernetes products imply the
  skill itself.
- `ci/cd`: `github actions` — a CI/CD system *is* CI/CD experience; now
  unambiguous after `github` left the `git` aliases.
- `c#`: `.net`, `dotnet`, `asp.net` — platform-as-language conflation accepted
  as the dominant resume convention.
- `machine learning`: `deep learning` — subset-implies-parent is safe.
- `go` has **no bare `go` alias** on purpose: the two-letter English word is
  unmatchable without false positives ("go to market"). `golang` is the
  reliable alias; a resume that only ever writes "Go" will be missed
  (documented false negative, visible in the eval).
- Bare `c` is likewise deliberately outside the dictionary.

## Limitations (deliberate, documented)

- **JD-side negation is not modelled**: "No Python required." in a JD still
  makes python a requirement (fixture `jd_side_negation_not_modelled`).
- **Years of experience, proficiency level, recency, and scope** ("not only X
  but also Y") are not modelled; a 10-year expert and a one-project user score
  identically on a matched keyword.
- Skills outside the compact dictionary are invisible to both sides.
- The negation filter is segment-scoped and pattern-based; unusual phrasings
  ("Kubernetes? Not for me") will not be caught.

## API configuration notes (private/full product path)

- `CORS_ALLOW_ORIGINS` — comma-separated CORS allowlist. **There is no
  wildcard**: unset falls back to `http://localhost:3000` /
  `http://127.0.0.1:3000` for local dev; production deployments must set it to
  the real frontend origin(s).
- `KAFKA_BOOTSTRAP_SERVERS` — enables the async analysis path. Unset (the
  default) makes `/analyze/async` and `/analyze/status/{id}` fail deliberately
  with `503 {"error": …, "code": "async_disabled"}`; there is no localhost
  fallback.
- `LLM_PROVIDER` / `OPENROUTER_API_KEY` — configure the optional LLM
  suggestion provider. Even when configured, resume/JD text is only sent when
  the request itself opts in with `"llm_consent": true`; default is the
  deterministic provider (legacy `LLM_PROVIDER=mock` still accepted and means
  deterministic). On provider failure the response falls back to deterministic
  suggestions and reports `provider: "deterministic (openrouter fallback)"`.
- `API_KEYS`, `RATE_LIMIT_PER_MINUTE` — optional auth/rate limiting (see
  `backend/app/security.py`).
- All error responses use the typed `{ "error": string, "code": string }`
  schema; all responses carry `Cache-Control: no-store`. `/extract` enforces a
  5 MB streamed upload cap, magic-byte checks (`%PDF-`, `PK`), a 30-page PDF
  bound and DOCX zip-bomb guards, and never logs filenames or content.
